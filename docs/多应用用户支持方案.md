# 多应用系统用户支持方案

## 一、当前设计的局限性

### 1.1 当前设计分析

**当前数据库设计**：
```sql
-- 用户表
CREATE TABLE `sys_user` (
    `app_code` VARCHAR(50) NOT NULL COMMENT '应用编码',
    `username` VARCHAR(64) NOT NULL COMMENT '用户名',
    UNIQUE KEY `uk_app_username` (`app_code`, `username`)
);

-- 角色表
CREATE TABLE `sys_role` (
    `app_code` VARCHAR(50) NOT NULL COMMENT '应用编码',
    `code` VARCHAR(100) NOT NULL COMMENT '角色编码',
    UNIQUE KEY `uk_app_role_code` (`app_code`, `code`)
);
```

**局限性**：
- ❌ 每个用户只能属于一个应用系统
- ❌ 用户无法在多个应用中使用
- ❌ 需要为每个应用创建独立的用户记录
- ❌ 用户数据冗余，维护困难

### 1.2 实际业务场景

**典型场景**：
1. **企业统一身份认证**：员工需要在OA、CRM、ERP等多个系统中使用同一账号
2. **SaaS平台**：用户需要在不同的SaaS应用中切换
3. **集团企业**：子公司系统需要共享集团用户数据

---

## 二、解决方案对比

### 2.1 方案一：用户应用关联表（推荐）

**设计思路**：
- 用户表不绑定app_code
- 创建用户应用关联表（sys_user_app）
- 用户可以在多个应用中存在，每个应用中有不同的角色

**优点**：
- ✅ 用户可以在多个应用中使用
- ✅ 保持数据隔离
- ✅ 灵活的角色分配
- ✅ 符合RBAC模型

**缺点**：
- ⚠️ 需要修改现有表结构
- ⚠️ 查询复杂度增加

### 2.2 方案二：统一用户中心

**设计思路**：
- 用户表不绑定app_code
- 通过用户角色关联表中的app_code来区分不同应用的角色
- 一个用户在多个应用中可以有不同的角色

**优点**：
- ✅ 用户统一管理
- ✅ 角色灵活分配
- ✅ 修改较小

**缺点**：
- ⚠️ 角色管理复杂
- ⚠️ 权限判断逻辑复杂

### 2.3 方案三：多租户隔离（当前方案）

**设计思路**：
- 每个应用独立的用户数据
- 通过app_code严格隔离

**优点**：
- ✅ 数据隔离性好
- ✅ 安全性高
- ✅ 实现简单

**缺点**：
- ❌ 用户无法跨应用
- ❌ 数据冗余
- ❌ 维护困难

## 三、推荐方案：用户应用关联表

### 3.1 数据库表设计

#### 修改后的表结构：

```sql
-- 1. 用户表（移除app_code约束）
CREATE TABLE `sys_user` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '用户ID',
    `username` VARCHAR(64) NOT NULL COMMENT '用户名（全局唯一）',
    `password` VARCHAR(128) NOT NULL COMMENT '密码',
    `nickname` VARCHAR(64) COMMENT '昵称',
    `email` VARCHAR(100) COMMENT '邮箱',
    `phone` VARCHAR(20) COMMENT '手机号',
    `avatar` VARCHAR(500) COMMENT '头像',
    `status` TINYINT DEFAULT 1 COMMENT '状态：0-禁用，1-正常',
    `is_super_admin` TINYINT DEFAULT 0 COMMENT '是否超级管理员：0-否，1-是',
    `last_login_time` DATETIME COMMENT '最后登录时间',
    `last_login_ip` VARCHAR(50) COMMENT '最后登录IP',
    `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_time` DATETIME DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    `create_by` BIGINT COMMENT '创建人',
    `update_by` BIGINT COMMENT '更新人',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_username` (`username`),
    UNIQUE KEY `uk_email` (`email`),
    UNIQUE KEY `uk_phone` (`phone`),
    KEY `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表（全局）';

-- 2. 用户应用关联表（新增）
CREATE TABLE `sys_user_app` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '关联ID',
    `user_id` BIGINT NOT NULL COMMENT '用户ID',
    `app_code` VARCHAR(50) NOT NULL COMMENT '应用编码',
    `dept_id` BIGINT COMMENT '部门ID（在该应用中的部门）',
    `status` TINYINT DEFAULT 1 COMMENT '状态：0-禁用，1-正常',
    `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `create_by` BIGINT COMMENT '创建人',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_user_app` (`user_id`, `app_code`),
    KEY `idx_user_id` (`user_id`),
    KEY `idx_app_code` (`app_code`),
    KEY `idx_dept_id` (`dept_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户应用关联表';

-- 3. 用户角色关联表（增加app_code）
CREATE TABLE `sys_user_role` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '关联ID',
    `user_id` BIGINT NOT NULL COMMENT '用户ID',
    `role_id` BIGINT NOT NULL COMMENT '角色ID',
    `app_code` VARCHAR(50) NOT NULL COMMENT '应用编码（角色所属应用）',
    `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `create_by` BIGINT COMMENT '创建人',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_user_role_app` (`user_id`, `role_id`, `app_code`),
    KEY `idx_user_id` (`user_id`),
    KEY `idx_role_id` (`role_id`),
    KEY `idx_app_code` (`app_code`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户角色关联表';

-- 4. 角色表（保持不变）
CREATE TABLE `sys_role` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '角色ID',
    `app_code` VARCHAR(50) NOT NULL COMMENT '应用编码',
    `name` VARCHAR(100) NOT NULL COMMENT '角色名称',
    `code` VARCHAR(100) NOT NULL COMMENT '角色编码',
    `data_scope` TINYINT DEFAULT 1 COMMENT '数据权限范围',
    `sort` INT DEFAULT 0 COMMENT '排序',
    `status` TINYINT DEFAULT 1 COMMENT '状态：0-禁用，1-正常',
    `remark` VARCHAR(500) COMMENT '备注',
    `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_time` DATETIME DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    `create_by` BIGINT COMMENT '创建人',
    `update_by` BIGINT COMMENT '更新人',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_app_role_code` (`app_code`, `code`),
    KEY `idx_app_code` (`app_code`),
    KEY `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='角色表';

-- 5. 权限表（保持不变）
CREATE TABLE `sys_permission` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '权限ID',
    `app_code` VARCHAR(50) NOT NULL COMMENT '应用编码',
    `parent_id` BIGINT DEFAULT 0 COMMENT '父权限ID',
    `name` VARCHAR(100) NOT NULL COMMENT '权限名称',
    `code` VARCHAR(100) NOT NULL COMMENT '权限编码',
    `type` TINYINT NOT NULL COMMENT '权限类型：1-菜单，2-按钮，3-接口，4-数据',
    `path` VARCHAR(255) COMMENT '路由路径/接口路径',
    `component` VARCHAR(255) COMMENT '前端组件',
    `method` VARCHAR(10) COMMENT 'HTTP方法',
    `icon` VARCHAR(100) COMMENT '图标',
    `sort` INT DEFAULT 0 COMMENT '排序',
    `status` TINYINT DEFAULT 1 COMMENT '状态：0-禁用，1-正常',
    `description` VARCHAR(500) COMMENT '描述',
    `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_time` DATETIME DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    `create_by` BIGINT COMMENT '创建人',
    `update_by` BIGINT COMMENT '更新人',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_app_permission_code` (`app_code`, `code`),
    KEY `idx_app_code` (`app_code`),
    KEY `idx_parent_id` (`parent_id`),
    KEY `idx_type` (`type`),
    KEY `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='权限资源表';

-- 6. 部门表（保持不变）
CREATE TABLE `sys_dept` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '部门ID',
    `app_code` VARCHAR(50) NOT NULL COMMENT '应用编码',
    `name` VARCHAR(100) NOT NULL COMMENT '部门名称',
    `parent_id` BIGINT DEFAULT 0 COMMENT '父部门ID',
    `ancestors` VARCHAR(500) COMMENT '祖级列表，逗号分隔',
    `leader` VARCHAR(50) COMMENT '负责人',
    `phone` VARCHAR(20) COMMENT '联系电话',
    `email` VARCHAR(100) COMMENT '邮箱',
    `sort` INT DEFAULT 0 COMMENT '排序',
    `status` TINYINT DEFAULT 1 COMMENT '状态：0-禁用，1-正常',
    `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_time` DATETIME DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    `create_by` BIGINT COMMENT '创建人',
    `update_by` BIGINT COMMENT '更新人',
    PRIMARY KEY (`id`),
    KEY `idx_app_code` (`app_code`),
    KEY `idx_parent_id` (`parent_id`),
    KEY `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='部门表';
```

### 3.2 数据关系图

```
sys_user (全局用户)
    ↓
sys_user_app (用户应用关联)
    ↓ app_code
sys_dept (应用部门)
    ↓
sys_user_role (用户角色关联)
    ↓ app_code
sys_role (应用角色)
    ↓
sys_role_permission (角色权限关联)
    ↓
sys_permission (应用权限)
```

### 3.3 使用示例

#### 场景：用户张三在OA和CRM两个系统中使用

```sql
-- 1. 创建全局用户
INSERT INTO sys_user (username, password, nickname, email, phone, status)
VALUES ('zhangsan', '$2a$10$...', '张三', 'zhangsan@example.com', '13800138000', 1);

-- 2. 关联到OA系统
INSERT INTO sys_user_app (user_id, app_code, dept_id, status)
VALUES (1, 'oa', 1, 1);

-- 3. 关联到CRM系统
INSERT INTO sys_user_app (user_id, app_code, dept_id, status)
VALUES (1, 'crm', 2, 1);

-- 4. 在OA系统中分配角色
INSERT INTO sys_user_role (user_id, role_id, app_code)
VALUES (1, 1, 'oa');  -- OA管理员角色

-- 5. 在CRM系统中分配角色
INSERT INTO sys_user_role (user_id, role_id, app_code)
VALUES (1, 2, 'crm');  -- CRM普通用户角色
```

### 3.4 权限查询逻辑

```sql
-- 查询用户在指定应用中的权限
SELECT DISTINCT p.*
FROM sys_permission p
INNER JOIN sys_role_permission rp ON p.id = rp.permission_id
INNER JOIN sys_role r ON rp.role_id = r.id
INNER JOIN sys_user_role ur ON r.id = ur.role_id
INNER JOIN sys_user_app ua ON ur.user_id = ua.user_id AND ur.app_code = ua.app_code
WHERE ua.user_id = #{userId}
  AND ua.app_code = #{appCode}
  AND ua.status = 1
  AND r.status = 1
  AND p.status = 1;
```

## 四、迁移方案

### 4.1 数据迁移步骤

```sql
-- 1. 创建新表
-- 执行上述表结构创建语句

-- 2. 迁移用户数据
INSERT INTO sys_user_new (id, username, password, nickname, email, phone, avatar, status, is_super_admin, last_login_time, last_login_ip, create_time, update_time, create_by, update_by)
SELECT id, username, password, nickname, email, phone, avatar, status, is_super_admin, last_login_time, last_login_ip, create_time, update_time, create_by, update_by
FROM sys_user;

-- 3. 创建用户应用关联
INSERT INTO sys_user_app (user_id, app_code, dept_id, status, create_time)
SELECT id, app_code, dept_id, status, create_time
FROM sys_user;

-- 4. 更新用户角色关联
INSERT INTO sys_user_role_new (user_id, role_id, app_code, create_time, create_by)
SELECT ur.user_id, ur.role_id, r.app_code, ur.create_time, ur.create_by
FROM sys_user_role ur
INNER JOIN sys_role r ON ur.role_id = r.id;

-- 5. 重命名表
RENAME TABLE sys_user TO sys_user_old;
RENAME TABLE sys_user_new TO sys_user;
RENAME TABLE sys_user_role TO sys_user_role_old;
RENAME TABLE sys_user_role_new TO sys_user_role;
```

### 4.2 代码修改

#### UserDetailsServiceImpl.java

```java
@Override
public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
    // 查询全局用户
    SysUser user = userMapper.selectOne(
        new QueryWrapper<SysUser>().eq("username", username)
    );
    
    if (user == null) {
        throw new UsernameNotFoundException("用户不存在");
    }
    
    // 查询用户应用关联
    List<SysUserApp> userApps = userAppMapper.selectList(
        new QueryWrapper<SysUserApp>().eq("user_id", user.getId()).eq("status", 1)
    );
    
    // 查询用户权限（需要指定app_code）
    List<SysPermission> permissions = permissionMapper.selectPermissionsByUserIdAndAppCode(
        user.getId(), 
        appCode  // 从请求头或参数中获取
    );
    
    return new AuthUser(user, userApps, permissions);
}
```

## 五、总结

### 5.1 方案对比

| 方案 | 用户跨应用 | 数据隔离 | 实现复杂度 | 推荐度 |
|------|-----------|----------|-----------|--------|
| 用户应用关联表 | ✅ 支持 | ✅ 良好 | ⭐⭐⭐ 中等 | ⭐⭐⭐⭐⭐ |
| 统一用户中心 | ✅ 支持 | ⚠️ 一般 | ⭐⭐ 简单 | ⭐⭐⭐⭐ |
| 多租户隔离（当前） | ❌ 不支持 | ✅ 优秀 | ⭐ 简单 | ⭐⭐ |

### 5.2 推荐方案

**推荐使用"用户应用关联表"方案**，理由：
- ✅ 支持用户在多个应用中使用
- ✅ 保持良好的数据隔离
- ✅ 灵活的角色分配机制
- ✅ 符合企业级应用需求
- ✅ 易于扩展和维护

### 5.3 实施建议

1. **评估业务需求**：确认是否真的需要用户跨应用

3. **修改代码**：更新Service层和Controller层逻辑
